OAuth2 Provider

Generate and introspect OAuth2 tokens in WASL Portal runtime.

WASL includes OAuth2 provider capabilities for client credentials flow. Use it to issue access tokens for app credentials and validate those tokens through gateway auth.oauth2 policies.

Endpoints

  • POST /oauth/token (canonical)
  • POST /oauth/introspect (canonical)
  • Compatibility aliases: /portal/oauth/token and /portal/oauth/introspect

Runtime host: http://localhost:9091 in local development.

Provider management APIs (Studio control plane)

OAuth provider profiles are configured from Studio and consumed by gateway policy binding.

  • GET/POST /api/security/oauth-providers
  • GET/PUT/DELETE /api/security/oauth-providers/{providerId}
  • POST /api/security/oauth-providers/{providerId}/test
  • POST /api/security/oauth-providers/{providerId}/promote
  • GET/POST /api/security/oauth-provider-credentials

Provider health telemetry is exposed through GET /api/analytics/auth/providers.

Issue Token (client_credentials)

curl -X POST "http://localhost:9091/oauth/token" \
  -u "wasl_client_xxx:wasl_secret_xxx" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=client_credentials&scope=orders.read"

Expected response

{
  "access_token": "wasl_oat_...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "scope": "orders.read",
  "app_id": "app_...",
  "tenant_id": "tenant_...",
  "client_id": "wasl_client_..."
}

Introspect Token

curl -X POST "http://localhost:9091/oauth/introspect" \
  -u "wasl_client_xxx:wasl_secret_xxx" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "token=wasl_oat_xxx"

Expected response

{
  "active": true,
  "tenant_id": "tenant_...",
  "scope": "orders.read",
  "token_type": "Bearer",
  "exp": 1775142630,
  "app_id": "app_...",
  "client_id": "wasl_client_...",
  "username": "dev_..."
}

Invalid/expired token returns {"active": false}.

Gateway Policy Wiring

{
  "type": "auth.oauth2",
  "providerRef": "wasl-internal-dev",
  "validationMode": "internal",
  "requiredScopes": ["orders.read"],
  "scopeMatchMode": "all",
  "audienceMode": "provider_default",
  "enforceProductPlan": true,
  "allowAnonymous": false
}

In enterprise setups, bind each API policy to a provider profile with providerRef. Do not place static client credentials inside API policy JSON.

WASL Internal Provider profile

When your token issuer is WASL internal OAuth (/oauth/token), configure provider profile as:

  • providerType = wasl_internal
  • validationMode = internal
  • clientAuthMethod = none
  • authCredentialRef = null

This avoids static introspection credential wiring for app-generated OAuth clients.

Runtime authorization model

Successful OAuth validation alone is not enough. WASL then enforces app subscription and plan limits:

  1. Credential must be active for a consumer app.
  2. App must be subscribed to a product containing the target operation.
  3. Plan scopes and limits must allow the request.

When limits are exceeded, gateway returns 429 Plan rate limit exceeded.

Mixed-provider validation (recommended)

  1. Create two provider profiles in Studio: one for jwks, one for introspection.
  2. Bind APIs to specific providers using providerRef.
  3. Verify wrong-provider tokens are denied (no cross-provider fallback).
  4. Verify plan + operation scopes still enforce after token validation.

  • docs/testing/CONSUMER_ACCESS_E2E_GUIDE.md (master end-to-end flow)

Verification order

Always use the same token value across token generation, introspection, and gateway API call. Most false failures are caused by testing different token values.
