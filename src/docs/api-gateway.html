Product · Unified Gateway

Build APIs, integrations, and AI routes on one gateway model

The WASL Unified Gateway is the data-plane runtime for API routes, FlowJS integration routes, plugin-backed backend calls, event listener flows, AI routes, policies, revisioned deployments, tenant isolation, and distributed tracing.

Key concepts

  • Bundle: a versioned workspace containing APIs, FlowJS flows, targets, plugin references, triggers, and policies.
  • Route: the public entry point such as GET /orders, an AI route, or a listener-triggered integration.
  • FlowJS: JavaScript orchestration logic for request shaping, transforms, target calls, and plugin adapter calls.
  • Gateway plugin: installable connectivity or governance extension invoked from FlowJS or the AI governance chain.
  • Policy: auth, security, traffic, validation, governance, or response control applied around the route and target calls.
Unified execution path
1
Ingress

Client, partner, app, scheduler, event listener, or AI client enters through a tenant and environment route.

2
Gateway + Policies

Authentication, validation, security, quota, and traffic controls run before FlowJS or provider execution.

3
FlowJS + Plugins

FlowJS calls HTTP targets and gateway plugins for databases, events, search, storage, custom systems, or AI providers.

4
Response + Trace

Gateway returns the result and records policy events, flow steps, plugin calls, payload samples, and correlation IDs.

Typical lifecycle

  1. Create a bundle for the API, integration, or AI surface.
  2. Add a public route, listener trigger, or AI route.
  3. Implement FlowJS logic or configure the AI governance chain.
  4. Install and configure gateway plugins per tenant and environment.
  5. Attach policies around ingress, target/plugin calls, and responses.
  6. Deploy the immutable revision to an environment.
  7. Package operations in a consumer product and subscribe apps.
  8. Promote or roll back as needed.

Runtime authorization and quota model

For enterprise access control, WASL evaluates request identity and subscription together:

  • auth.apikey or auth.oauth2 validates credential identity.
  • Gateway resolves app subscription to product operation scope.
  • Gateway enforces plan scopes and rate/quota limits.

This keeps docs exposure and runtime authorization aligned across Studio, Portal, and Gateway.

Validation guide: docs/testing/CONSUMER_ACCESS_E2E_GUIDE.md.

KVM and runtime config

KVM is a gateway-side configuration store used for runtime lookups and secret references. It is managed in Studio and read inside FlowJS via flow.kvm or the Apigee-compatible context API.

  • Use it for tenant/environment-scoped values.
  • Do not inject KVM data into request payloads.
  • Read values with context.getVariable('kvm.<contextKey>') when you need a direct lookup.

Targets, plugins, and outbound calls

Use targets for HTTP systems and gateway plugins for databases, queues, search, storage, and custom services. Both are environment-aware, so the same FlowJS code can run across dev, staging, and production.

const result = http.getJson('erp-service', {
  path: '/orders',
  query: { status: 'OPEN' }
});
