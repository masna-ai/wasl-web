API Gateway

Build and govern API routes

WASL API Gateway focuses on route definition, policies, revisioned deployments, targets, and runtime observability.

Key concepts

  • Bundle: a versioned workspace containing APIs, flows, and policies.
  • API: the public route definition such as GET /orders.
  • Flow: the runtime logic executed for the API.
  • Policy: auth, security, or traffic control applied to the route.
Request Path
1
Client Request

Client calls the tenant and environment route through the public gateway hostname.

2
Gateway + Policies

Authentication, security, and traffic controls execute before any flow code runs.

3
Flow + Targets

Flow runtime calls targets, adapters, databases, or other systems using environment-aware configuration.

4
Response + Trace

Gateway returns the response and records a traceable execution path for debugging and operations.

Typical lifecycle

  1. Create bundle.
  2. Add API endpoint.
  3. Implement flow logic.
  4. Attach policies.
  5. Deploy revision to environment.
  6. Package operations in a consumer product and subscribe apps.
  7. Promote or roll back as needed.

Runtime authorization and quota model

For enterprise access control, WASL evaluates request identity and subscription together:

  • auth.apikey or auth.oauth2 validates credential identity.
  • Gateway resolves app subscription to product operation scope.
  • Gateway enforces plan scopes and rate/quota limits.

This keeps docs exposure and runtime authorization aligned across Studio, Portal, and Gateway.

Validation guide: docs/testing/CONSUMER_ACCESS_E2E_GUIDE.md.

KVM and runtime config

KVM is a gateway-side configuration store used for runtime lookups and secret references. It is managed in Studio under API Gateway → KVM, then read inside flow scripts via flow.kvm.

  • Use it for tenant/environment-scoped values.
  • Do not inject KVM data into request payloads.
  • Read values with context.getVariable('kvm.<contextKey>') when you need a direct lookup.

Targets and outbound calls

Use targets when your flow needs to call external HTTP systems. Targets are environment-aware and designed to separate logical route references from actual endpoint configuration.

const result = http.getJson('erp-service', {
  path: '/orders',
  query: { status: 'OPEN' }
});
Public Route What clients call through the gateway.
Bundle Target The logical alias referenced by the flow.
Environment Target The actual endpoint configured per environment.
