Adapters

Extend runtime connectivity

Adapters let flows connect to systems such as databases, messaging platforms, or custom runtime extensions without hardwiring those capabilities into every route.

Common usage

  • JDBC for relational data access.
  • Kafka for event production and consumption.
  • Marketplace-installed adapters for standardized integration patterns.

How adapters fit into runtime

Adapters are configured per tenant and per environment. Flows call adapters by alias, while the actual connection details stay in the environment configuration layer.

JDBC example

Use the WASL JDBC Adapter for relational database access from Flow JS.

const rows = adapters.queryJson('wasl-jdbc', {
  connection: 'default',
  sql: 'SELECT id, name FROM customers'
});

response.status = 200;
response.body = {
  count: rows.length,
  data: rows
};

Typical JDBC configuration is stored per environment and includes the JDBC URL, credentials, driver class, driver jar path, and validation query.

Kafka example

Use Kafka adapters for event production and event-driven consumption.

adapters.invoke('kafka', {
  connection: 'kafka',
  operation: 'produce',
  topic: 'customer-events',
  key: flow.utils.uuid(),
  payload: JSON.stringify({ orderId: 42, status: 'shipped' })
});

Kafka listeners are deployment-aware. They run only for deployed revisions and can auto-start or auto-stop as deployments change.

Troubleshooting

  • If JDBC validation fails, verify the URL starts with jdbc:, the driver path is valid, and simulate is disabled for real checks.
  • If Kafka produce succeeds but no message appears, confirm the broker endpoint matches the network path used by WASL.
  • If an adapter is reported as not installed, verify the plugin scope, tenant installation, and environment config.
