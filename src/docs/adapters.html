Gateway Plugins

iPaaS connectivity, as gateway plugins

Gateway plugins extend WASL with integration connectivity and runtime capabilities: databases, messaging, search, object storage, custom systems, and AI governance controls. FlowJS calls integration plugins by alias, while Studio manages installation, tenant scope, environment config, versioning, and connection tests.

Built-in integration plugins

  • JDBC — PostgreSQL, MySQL, Oracle, SQLite
  • Kafka — producer / consumer with retry and DLQ
  • MongoDB, Redis, RabbitMQ
  • OpenSearch — index, search, aggregations
  • Object Storage (S3-compatible)
  • HTTP — via fleet-resolved target servers (same flow code runs in dev, staging, and prod)

Custom plugins can be authored against the public Java SPI and distributed through the Plugin Marketplace. The same platform model also hosts AI governance plugins such as auth, guardrails, PII handling, semantic cache, prompt management, quota, cost, retry, and provider routing.

How plugins fit into runtime

Plugins are configured per tenant and per environment. FlowJS invokes integration plugins by alias through the adapters binding, while actual credentials, hosts, topics, buckets, indexes, and driver settings stay in the environment configuration layer.

Plugin invocation model
FlowJS script Calls adapters.query, adapters.update, or adapters.invoke by alias.
Plugin registry Resolves tenant/global installation, plugin version, descriptor, and active classloader.
Environment config Applies dev/staging/prod connection settings and secrets before executing the plugin call.

JDBC example

Use the WASL JDBC plugin for relational database access from FlowJS. Reads use query; writes should use update or pass operation: 'update'.

const rows = adapters.queryJson('wasl-jdbc', {
  connection: 'default',
  sql: 'SELECT id, name FROM customers'
});

response.status = 200;
response.body = {
  count: rows.length,
  data: rows
};

Typical JDBC configuration is stored per environment and includes the JDBC URL, credentials, driver class, driver jar path, and validation query.

Kafka example

Use the Kafka plugin for event production and event-driven consumption. Listener flows are generated and managed through plugin configuration, then run only for deployed revisions.

adapters.invoke('kafka', {
  connection: 'kafka',
  operation: 'produce',
  topic: 'customer-events',
  key: flow.utils.uuid(),
  payload: JSON.stringify({ orderId: 42, status: 'shipped' })
});

Kafka listeners are deployment-aware. They run only for deployed revisions and can auto-start or auto-stop as deployments change.

Plugin operating model

  • Install a plugin globally or for a tenant through Studio or the marketplace APIs.
  • Configure environment-specific connections for dev, staging, production, and custom environments.
  • Test the connection from the plugin detail page before routing traffic through it.
  • Invoke the plugin from FlowJS by alias; scripts do not hold credentials.
  • Upgrade plugins with classloader eviction and active-call draining so new calls use fresh plugin code.

Troubleshooting

  • If JDBC validation fails, verify the URL starts with jdbc:, the driver path is valid, and simulate is disabled for real checks.
  • If Kafka produce succeeds but no message appears, confirm the broker endpoint matches the network path used by WASL.
  • If a plugin is reported as not installed, verify the plugin scope, tenant installation, and environment config.
