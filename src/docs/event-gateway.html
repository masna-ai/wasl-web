Product · Event Gateway

Pub/sub over HTTP — browsers, mobile, partners, edge.

The WASL Event Gateway exposes event streams as plain HTTP endpoints. Producers POST records to an endpoint, consumers stream over Server-Sent Events or long-poll, and offsets are managed by the gateway. Any client that speaks HTTP joins the same event flow your backend uses — no Kafka SDK, no broker credentials in third-party hands, no extra network paths to govern.

Key concepts

  • Endpoint: a named, governed entry point — for example order-events. Producers POST to it; consumers subscribe to it.
  • Contract: an optional JSON Schema attached to an endpoint. Records that fail validation are rejected at produce time, before they reach the broker.
  • Subscription: a durable position in the stream for a specific consumer. Survives reconnects, browser refreshes, and pod restarts.
  • Cluster: the underlying broker (Kafka today). Endpoints reference a cluster by name so you can move them between brokers without touching producers or consumers.
  • AsyncAPI document: auto-generated, downloadable spec describing every endpoint — feed it to clients the same way you feed OpenAPI to REST consumers.
End-to-end event flow
1
Produce

Client POSTs to /events/{endpoint}/records. Gateway authenticates, validates against the contract, then writes to the broker.

2
Subscribe

Consumer opens an SSE stream at /events/{endpoint}/stream or long-polls /subscriptions/{subId}/records.

3
Commit

Consumer POSTs the last-processed offset to /subscriptions/{subId}/commit — durable position in the stream.

4
Observe

Every produce, deliver, and commit emits a trace event with the same correlation ID format as REST and AI traffic.

Why HTTP for events

A traditional broker like Kafka assumes every client can run a long-lived SDK with cluster discovery, partition assignment, consumer groups, and offset management. That's fine for back-end services but a poor fit for:

  • Browsers and mobile apps — no Kafka SDK, no broker credentials at the edge.
  • Edge workers and serverless functions — cold-start hostile and short-lived.
  • Partner systems and webhooks — outbound HTTP is the only thing the perimeter allows.
  • Polyglot teams — every language has an HTTP client; not every language has a maintained Kafka client.

The Event Gateway sits in front of your broker and lets all of these clients participate in the same event streams as your back-end services, with the same authentication, quota, and trace pipeline as the rest of WASL.

Producing records

POST one or more records to the endpoint. The body is an array of JSON objects; the gateway validates each against the contract before forwarding to the broker.

POST /events/order-events/records
Authorization: Bearer <app credential>
Content-Type: application/json

[
  { "orderId": "ord_1234", "status": "PLACED", "amount": 49.95 },
  { "orderId": "ord_1235", "status": "PLACED", "amount": 12.00 }
]

Successful writes return per-record offsets so the producer can correlate them with downstream consume events. Contract violations are rejected with a 422 and a per-record reason — no partial writes.

Consuming records

Two delivery modes are supported on the same endpoint — pick whichever fits your client:

Server-Sent Events Long-lived GET /events/{endpoint}/stream. The gateway pushes records as they arrive. Ideal for browsers, dashboards, and any client that can hold an open connection.
Long-poll Repeated GET /subscriptions/{subId}/records. The gateway holds the request open until records are available or a timeout elapses. Ideal for serverless, mobile background fetch, and proxy-traversing clients.
Offset commit POST /subscriptions/{subId}/commit tells the gateway where to resume on the next consume. Subscriptions survive reconnects, refreshes, and pod restarts.

Governance — same as everything else

Event endpoints sit behind the unified policy engine. Attach the same primitives you'd attach to a REST route:

  • auth.apikey, auth.oauth2, or auth.jwt at produce and consume time.
  • rate-limit and quota per app, per tenant, or per endpoint.
  • data-masking and pii on payload fields before they hit the broker.
  • request-validation for stricter contracts than JSON Schema can express.

Subscription approvals, app-to-endpoint bindings, and per-plan quotas flow through the same Portal + Products + Subscriptions model used by REST and AI products.

AsyncAPI out of the box

Every endpoint has an auto-generated AsyncAPI 2.x document at GET /api/event-gateway/endpoints/{id}/asyncapi. It includes the contract schema, supported delivery modes, and ready-to-paste connection snippets (curl, Node, Python) so consumers can integrate without reading custom docs.

Same trace pipeline as the rest of WASL

A single correlation ID follows a request from the REST API edge, through a FlowJS handler, into a produce call on an event endpoint, out to a consuming worker, and into whatever downstream call that worker makes. One trace view, all traffic types.

What's coming

The current Event Gateway is HTTP/SSE-first and Kafka-backed. The schema reserves protocol_modes so a future release can add native Kafka protocol proxy for high-throughput back-end consumers without breaking the HTTP API. Disconnected / edge-buffered producers are explicitly v2+ and tracked separately.
