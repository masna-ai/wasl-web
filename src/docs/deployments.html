Deployments

Move safely across environments

WASL uses revisioned deployments so teams can deploy, promote, and roll back without losing the exact runtime state of each environment.

Deployment model

  • Development is the working source area.
  • Deployments bind a specific revision to an environment.
  • Promotion moves a tested revision into the next environment.
  • Rollback returns the environment to a previous deployed revision.
Revision Lifecycle
1
Working Revision

Develop and save changes in Studio to create a new revision for the bundle.

2
Deploy to Dev

The selected environment points to the chosen revision instead of copying source files.

3
Promote

Move the same tested revision into staging or production when it is ready.

4
Rollback

Re-point the environment to a previous deployed revision if you need to recover quickly.

Core concepts

  • Revision-based source: bundle revisions are stored under tenant-scoped revision folders.
  • Deployment pointers: environments point to a selected revision rather than copying files around.
  • Environment-specific configs: adapters and targets can differ per environment.

Environment resolution

WASL resolves the active environment in this order:

  1. X-Environment header
  2. Environment slug in the request host
  3. Path prefix such as /env/staging/...
  4. Default fallback to dev

Studio workflow

  1. Create or verify environments in the Environments workspace.
  2. Save your bundle to create a new revision.
  3. Deploy the chosen revision to the target environment.
  4. Promote between environments when the revision is ready.
  5. Roll back by pointing the environment to a previous revision if required.

Troubleshooting

  • If the wrong revision is served, verify the active deployment pointer for the tenant, bundle, and environment.
  • If an adapter is missing at runtime, confirm the adapter config exists for that environment.
  • If a target server is missing, verify the environment-specific target exists and matches the alias used in the flow.
