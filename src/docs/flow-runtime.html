FlowJS iPaaS Runtime

Orchestrate APIs, data, events, and systems in JavaScript

WASL FlowJS is the iPaaS execution layer inside the unified gateway. Flows run on GraalJS and can shape requests, validate payloads, call HTTP targets, invoke gateway plugins, publish events, load bundle-local modules, and emit traceable responses.

Where FlowJS fits

A FlowJS script is attached to a deployed route or trigger inside a bundle revision. The gateway resolves tenant, environment, bundle, revision, and policy context before the script runs, then records policy, target, plugin, and response events into the request trace.

Ingress logic Read and normalize request body, headers, query, path params, and auth context.
Integration logic Call targets and plugins for databases, queues, storage, search, and custom systems.
Response logic Shape output, set status, write headers, and return stable API contracts.

Runtime bindings

Flow scripts run with built-in objects such as request, response, input, context, flow, http, adapters, blocks, and console.

  • request.body gives parsed JSON when possible; prefer it over manual parsing.
  • http.getJson('target') calls 3-level environment-aware HTTP targets.
  • adapters.invoke('alias') calls installed gateway plugins by alias.
  • flow.require('resources/scripts/helper.js') loads reusable bundle-local modules.
  • flow.guard(fn) wraps integration logic with structured error handling.

Simple response example

response.status = 200;
response.setHeader('content-type', 'application/json');
response.body = {
  ok: true,
  runtime: 'wasl-flow-js'
};

Target call example

const result = http.getJson('erp-service', {
  path: '/orders',
  query: { status: 'OPEN' }
});

response.status = 200;
response.body = result;

Plugin call example

Gateway plugins are the connectivity layer for iPaaS flows. The script calls a stable alias; tenant and environment configuration resolve the actual connection.

const rows = adapters.queryJson('wasl-jdbc', {
  connection: 'default',
  sql: 'SELECT id, name FROM customers'
});

response.status = 200;
response.body = {
  count: rows.length,
  data: rows
};

Event-driven flow example

Plugins can also produce events or back listener-triggered flows. Listener lifecycle is deployment-aware, so production revisions stay immutable.

adapters.invoke('kafka', {
  connection: 'orders',
  operation: 'produce',
  topic: 'order-events',
  key: request.body.orderId,
  payload: JSON.stringify({
    orderId: request.body.orderId,
    status: 'accepted',
    correlationId: flow.correlationId
  })
});

response.status = 202;
response.body = { accepted: true };

FlowJS rules that matter

  • FlowJS runs as a program body. Do not use top-level return.
  • Keep business helpers under resources/scripts/ and load them with flow.require.
  • Put backend credentials in plugin or target environment configuration, not in scripts.
  • Use policies for reusable governance; use FlowJS for route-specific orchestration and transformation.
