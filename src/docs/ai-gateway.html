AI Gateway

Route and govern AI traffic

WASL AI Gateway lets you expose a stable route while managing provider selection, streaming, quotas, guardrails, and AI-specific controls behind it.

Supported provider model

Create an AI route, connect one or more providers, and configure the controls you need for that route.

  • OpenAI
  • Anthropic
  • Gemini
  • Ollama
AI Route Path
1
Client Route

Client calls a stable route such as POST /ai/openai-test.

2
AI Controls

WASL applies auth, guardrails, limits, quotas, prompt controls, and optional semantic cache.

3
Provider Routing

Routing chooses the model provider using priority, weighted, latency-aware, cost-aware, or health-aware strategies.

4
Response Path

Buffered or streaming output returns through the same governed route with analytics and tracing.

What you can configure

  • Authentication
  • Guardrails
  • Limits
  • Retry and circuit breaker
  • Quota and cost control
  • Prompt management
  • Semantic caching
  • RAG
  • PII sanitization
  • Load balancing strategies

Runtime endpoint pattern

AI routes are invoked through the gateway hostname for the active tenant and environment.

POST http://<tenant>.<environment>.localhost:8080/ai/{routeId}

Example:
POST http://acme-corp.dev.localhost:8080/ai/openai-test

Streaming

Clients control streaming behavior per request. WASL supports buffered responses and streaming responses from the same route.

  • raw for provider-native SSE passthrough.
  • normalized for stable WASL events such as token, usage, and done.
  • text for simple text-only SSE chunks.
curl -N -X POST http://acme-corp.dev.localhost:8080/ai/openai \
  -H 'Content-Type: application/json' \
  -H 'Accept: text/event-stream' \
  -d '{
    "stream": true,
    "streamMode": "normalized",
    "input": {
      "prompt": "Explain what WASL AI Gateway does"
    }
  }'

Typical route configuration

A typical AI route includes providers, a default routing choice, and plugin configs that define governance behavior.

{
  "id": "openai-test",
  "name": "OpenAI Test Route",
  "environment": "dev",
  "enabled": true,
  "defaultProvider": "openai-primary",
  "providers": [
    {
      "id": "openai-primary",
      "type": "openai",
      "model": "gpt-4o-mini",
      "apiKeyEnv": "OPENAI_API_KEY",
      "baseUrl": "https://api.openai.com/v1",
      "timeoutMs": 15000
    }
  ]
}

Operational guidance

  • Start with one provider, then add fallback or load-balancing strategies once the route is stable.
  • Use authentication and guardrails before enabling public or partner traffic.
  • Turn on semantic caching and quota controls for cost-sensitive routes.
  • Validate route behavior in Studio before deployment to the target environment.
Govern Attach auth, guardrails, and cost controls before wider rollout.
Optimize Use semantic cache and routing strategies to improve latency and cost.
Observe Use tracing and analytics to validate provider choice and runtime behavior.
