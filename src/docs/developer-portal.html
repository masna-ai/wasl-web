Developer Portal

Expose APIs through shared, tenant-scoped, or managed customer portals.

WASL supports one portal backend and frontend with multiple exposure models. You can run a shared portal for internal discovery, a tenant-scoped portal for one customer, or managed portals with their own branding, publications, and onboarding rules.

Portal modes

  • Shared portal: `http://localhost:3000/` for broad discovery across visible tenants.
  • Tenant-scoped portal: `http://localhost:3000/?tenant=<tenantId-or-slug>` for one-tenant catalog and onboarding.
  • Managed customer portal: `http://localhost:3000/?portal=<portalId-or-slug>` for a Studio-managed portal entity.

What a managed portal controls

  • Portal branding such as title, hero copy, colors, docs link, and support link.
  • Published consumer products exposed to that portal.
  • Onboarding mode: self-service API keys or approval-based access requests.
  • Portal-specific webhook notifications and admin contacts.

What the portal does not manage

The portal is for consumers. Platform configuration stays in Studio.

  • KVM / Secret References live in Studio → API Gateway → KVM.
  • Certificates and OAuth provider profiles are also managed in Studio.
  • Flow scripts read KVM values through flow.kvm and context.getVariable('kvm.<contextKey>').

Product-first access model

The enterprise model is product-first and app-first:

Tenant → Product → Plan → App → Subscription → Credential → Gateway

  • Products define allowed operations and default limits.
  • Apps are runtime consumer identities.
  • Credentials (API key or OAuth client) are issued per app.
  • Portal is a publishing/onboarding surface, not the owner of products.

Recommended SaaS model

For SaaS, keep the WASL control plane multi-tenant, but expose developer portals per customer or per audience. Use shared portal mode for internal platform teams, tenant-scoped mode for default customer exposure, and managed customer portals when you need branded or publication-specific experiences.

Current development workflow

Create the portal in Studio under Common → Portals, publish bundles, then open it with ?portal=<portalId-or-slug>.

OAuth2 Provider Quickstart

  1. Create a consumer product and set operation scope + default limits.
  2. Create an app and subscribe it to the product.
  3. Issue an OAuth client credential for that app.
  4. Generate an access token using POST /oauth/token.
  5. Validate token state using POST /oauth/introspect.
  6. Attach auth.oauth2 policy using providerRef and call with bearer token.
