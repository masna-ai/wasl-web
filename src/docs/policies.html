Policies

Apply governance consistently

Policies are how WASL applies auth, security, traffic, reliability, and AI-specific controls consistently across routes and environments.

Policy categories

  • Authentication such as API key, JWT, OAuth2, basic auth, and mTLS.
  • Security controls such as IP filtering, request size, and header limits.
  • Traffic and reliability controls such as rate limiting and circuit breaker.
  • AI controls such as guardrails, quotas, cost limits, PII sanitization, and prompt management.

How policies work

Policies execute before flow logic, so authentication and core governance happen at the gateway edge instead of deep inside your runtime code. Policies are versioned with the bundle revision and move through environments with the rest of the deployment.

Common auth patterns

API Key

[
  {
    "type": "apikey",
    "headerName": "x-api-key",
    "apiKeys": ["wasl-dev-key-1", "wasl-dev-key-2"]
  }
]

JWT

[
  {
    "type": "jwt",
    "secret": "replace-with-shared-secret",
    "algorithm": "HS256",
    "requireExp": true,
    "clockSkewSeconds": 30
  }
]

mTLS

[
  {
    "type": "mtls",
    "enforceClientCert": true
  }
]

Operational guidance

  • Apply security policies before flow logic so requests fail early.
  • Attach policies at the API level and validate them in the deployed environment, not just in Studio.
  • For AI routes, combine authentication with guardrails, limits, quotas, and cost controls.
  • Use tracing when validating policy behavior so you can confirm which policy blocked or changed the request.
