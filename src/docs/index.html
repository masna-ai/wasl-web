WASL Documentation

WASL is a unified gateway platform: APIs, AI workloads, event streams, integration flows, and integration plugins on one runtime, governed by one policy engine, observed by one trace pipeline. These docs are written for end users, platform teams, and integration builders.

What WASL is

WASL gives platform teams one place to govern API traffic, AI traffic, event streams, and integration flows. Instead of running a separate gateway product, iPaaS runtime, event broker frontend, AI control layer, portal, and observability stack, you operate one platform with a shared policy engine, plugin SDK, FlowJS runtime, and deployment model.

Five surfaces

  • API Gateway — REST, HTTP/2, WebSocket, and gRPC routes with tenant-aware routing, policy enforcement, multi-tenancy, and versioned deployments.
  • AI Gateway — OpenAI-compatible LLM routing with guardrails, quotas, streaming, semantic caching, prompt versioning, PII controls, and cost caps.
  • Event Gateway — HTTP produce, SSE / long-poll consume, contracts, subscriptions, and AsyncAPI docs — pub/sub for clients that can't run a Kafka SDK.
  • FlowJS Runtime — JavaScript orchestration for request handling, transformation, HTTP calls, plugin calls, and reusable modules.
  • Gateway Plugins — connectors for JDBC, Kafka, MongoDB, Redis, RabbitMQ, OpenSearch, S3-compatible storage, HTTP, and custom systems via the public SDK.

Open source forever

How the product is organized

End users mainly work through the Studio control plane. Studio is where you create bundles, APIs, FlowJS flows, policies, targets, plugin connections, AI routes, listeners, products, and deployments. The gateway runtime then serves traffic based on those deployed assets.

Typical workflow

  1. Create an API route, AI route, event endpoint, or FlowJS integration in Studio.
  2. Install and configure the required plugins for each tenant and fleet.
  3. Attach policies and configure runtime behavior.
  4. Test the route, endpoint, or flow in Studio.
  5. Deploy to a fleet such as dev or staging.
  6. Invoke the route through the gateway hostname for the selected tenant and fleet.
