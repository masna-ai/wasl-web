WASL Documentation

WASL is a unified gateway and iPaaS platform: API governance, AI governance, FlowJS orchestration, and integration plugins in one deployment model. These docs are written for end users, platform teams, and integration builders.

Start here

Use the Quickstart to create your first API route, AI route, and deployment in minutes.

Unified gateway

The Gateway handles API routes, AI routes, FlowJS integrations, policies, deployments, and traces through one platform model.

Plugin-based iPaaS

Gateway Plugins provide backend, event, data, and custom system connectivity for FlowJS without bolting drivers into the gateway core.

What WASL is

WASL gives platform teams one place to govern API traffic, AI traffic, and integration flows. Instead of running a separate gateway product, iPaaS runtime, AI control layer, portal, and observability stack, you operate one platform with a shared policy engine, plugin SDK, FlowJS runtime, and deployment model.

Four pillars

  • Unified Gateway — API routes, AI routes, tenant-aware routing, policy enforcement, multi-tenancy, and revisioned deployments.
  • FlowJS Runtime — JavaScript orchestration for request handling, transformation, HTTP calls, plugin calls, and reusable bundle-local modules.
  • Gateway Plugins — backend connectivity for JDBC, Kafka, MongoDB, Redis, RabbitMQ, OpenSearch, S3-compatible storage, HTTP, and custom systems.
  • AI Governance — LLM routing, guardrails, quotas, streaming, semantic caching, prompt governance, PII controls, and cost caps.

Open source forever

Apache 2.0 licensed. Self-host on your own infrastructure. The full feature set — not a stripped-down community edition.

How the product is organized

End users mainly work through the Studio control plane. Studio is where you create bundles, APIs, FlowJS flows, policies, targets, plugin connections, AI routes, listeners, products, and deployments. The gateway runtime then serves traffic based on those deployed assets.

Typical workflow

  1. Create an API route, AI route, or FlowJS integration in Studio.
  2. Install and configure the required plugins for each tenant and environment.
  3. Attach policies and configure runtime behavior.
  4. Test the route or flow in Studio.
  5. Deploy to an environment such as dev or staging.
  6. Invoke the route through the gateway hostname for the selected tenant and environment.
