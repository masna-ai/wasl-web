WASL Documentation

WASL is the unified open-source API Gateway and AI Gateway with integrations as plugins. These docs are written for end users — not internal engineering teams.

Start here

Use the Quickstart to create your first API route, AI route, and deployment in minutes.

Two gateways, one platform

The API Gateway and AI Gateway are twin first-class products on a shared control and data plane.

Pluggable everywhere

Integration Plugins extend the platform — backend connectivity ships as plugins, not bolted into the core.

What WASL is

WASL gives platform teams one place to govern API traffic and AI traffic. Instead of running a separate gateway product, an AI control layer, and an integration runtime, you operate one open platform with a shared policy engine, plugin SDK, and deployment model.

Three pillars

  • API Gateway — routing, policy enforcement, multi-tenancy, revisioned deployments.
  • AI Gateway — LLM routing, guardrails, quotas, streaming, semantic caching, prompt governance, cost caps.
  • Integration Plugins — backend connectivity (JDBC, Kafka, MongoDB, Redis, RabbitMQ, OpenSearch, S3, HTTP) via a unified plugin SDK with marketplace distribution.

Open source forever

Apache 2.0 licensed. Self-host on your own infrastructure. The full feature set — not a stripped-down community edition.

Recommended path

If you are new to the platform, go to Quickstart first. It is the fastest way to understand the product model.

How the product is organized

End users mainly work through the Studio control plane. Studio is where you create bundles, APIs, flows, policies, targets, adapters, AI routes, and deployments. The gateway runtime then serves traffic based on those deployed assets.

Typical workflow

  1. Create an API route or AI route in Studio.
  2. Attach policies and configure runtime behavior.
  3. Test in Studio.
  4. Deploy to an environment such as dev or staging.
  5. Invoke the route through the gateway hostname for the selected tenant and environment.
