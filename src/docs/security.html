Security

Secure the platform and traffic path

Security in WASL spans route policies, tenant isolation, certificate management, and environment-aware runtime control.

Security topics

  • Authentication policies such as API key, JWT, OAuth2, and mTLS.
  • Certificate management for trust anchors and client certificates.
  • Tenant and environment isolation across control plane and data plane.
  • AI-specific controls such as guardrails and PII sanitization.

Recommended security model

For most teams, the right sequence is: authenticate first, apply request security controls next, then run flow or AI route logic. WASL executes these controls before flow execution so failures happen early and consistently.

Studio-managed security assets

  • Certificates are managed in Studio and used for mTLS and outbound TLS profiles.
  • KVM / Secret References are managed in Studio → API Gateway → KVM.
  • Both are consumed by gateway policies and flow runtime, not by the public portal.

Common authentication patterns

  • API key for partner and developer access.
  • JWT for shared-secret internal systems.
  • OAuth2 provider profiles (jwks, introspection, or hybrid) for identity-provider-backed tokens.
  • mTLS for B2B integrations that require client certificates.

For WASL-native token APIs and provider-based gateway wiring, see OAuth2 Provider API.

mTLS deployment pattern

WASL commonly runs behind an HTTPS front proxy such as Nginx. The proxy terminates TLS, validates the client certificate, and forwards the validated certificate to the gateway through the x-client-cert header for the auth.mtls policy.

curl https://acme-corp.dev.localhost:8443/crm/tls/test \
  --cert client.crt \
  --key client.key \
  --cacert certs/ca.crt

Operational guidance

  • Upload trust anchors and client key pairs through the Certificates workspace instead of hardwiring PEM material into route logic.
  • Validate security behavior in the deployed environment, not only in Studio tests.
  • For AI routes, combine auth with guardrails, quotas, and PII controls before opening routes to wider traffic.
  • Use tracing and analytics when validating policy behavior so you can see which control accepted or blocked the request.
