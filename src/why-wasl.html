Architectural Philosophy

Consolidate your stack. Run at the edge.

Traditional enterprise architectures force platform teams to run a different proxy engine for every kind of traffic. WASL breaks this pattern by running API management, AI governance, event-streaming, and background scheduling on a single, lightweight runtime.

The Challenge

The Cost of Tool Sprawl

As organizations adopt APIs, event-driven architectures, and LLMs, their network perimeter fractures. Operating multiple disparate runtimes leads to severe platform challenges:

01

Observability Gaps

Scattered logs, metrics, and traces across independent API management, AI proxies, and event streams make distributed debugging nearly impossible.

02

Policy Duplication

Rewriting authentication, rate-limiting, and validation rules in four separate systems introduces security risks and policy drift.

03

High Cloud Overhead

Operating multiple gateway clusters (often written in different languages and frameworks) inflates container counts and runtime resource requirements.

Pillars of WASL

Built from First-Principles

We engineered WASL to provide enterprise-grade capabilities with a developer-first aesthetic and minimal infrastructure footprint.

Jetty 12 + Project Loom Virtual Threads

Traditional gateways block native OS threads on slow downstream calls, causing severe memory overhead under load. Reactive gateways (Node, Netty) prevent blocking but force developers into complex reactive programming models.

Virtual Threading: Millions of concurrent requests mapped to lightweight JVM virtual threads Synchronous Syntax: Write straightforward, readable code without async/await callbacks Memory Efficiency: Scale to heavy load using only a fraction of the memory of legacy JVM servers

GraalJS-compiled Flow Engine

WASL integrates a high-performance JavaScript execution engine (Flow v3) powered by GraalVM. Your customization scripts run at native compiled speeds.

Polyglot Performance: JavaScript flows compiled directly to native Java bytecode Zero-overhead Proxying: Routes that do not require scripting run at pure hardware-proxy speeds Type Safety: Enforce strong typing, parameters, and policies before execution

Strict Control & Data Plane Isolation

The WASL data-plane gateway has zero runtime dependency on the WASL Studio control plane. Gateway runtimes are completely stateless and pull configurations asynchronously from Postgres or Redis.

Resiliency: Studio control plane outages have zero impact on live traffic Immutable Revisions: All configuration deployments are transactional, versioned database updates Horizontal Scale: Easily spin up gateways at the edge, on-prem, or in Kubernetes clusters

Pluggable Integration SDK

Instead of bundling every database and queue driver into the gateway core, WASL leverages a dynamic Java SPI plugin architecture.

Lean Footprint: The base core is less than 50MB, making it extremely fast to boot and deploy Dynamic Marketplace: Download signed adapters (JDBC, Kafka, Redis) on-demand Extension SDK: Author custom protocols and custom security adapters in standard Java
