Why WASL

Three things you won't find in a typical gateway.

Most platforms force a trade-off: a fast gateway with no integration runtime, or a flexible iPaaS that can't serve production API traffic. WASL refuses the trade-off — and treats AI as a first-class workload, not an afterthought plugin.

01

One runtime, three workloads

API management, integration orchestration, and AI governance share a single data plane — same policy engine, same trace pipeline, same revision model. Operate one system instead of three.

02

Execution-mode spectrum

Pick the cheapest path per route: direct-proxy for pass-through, flow for custom JavaScript, mock for contract-first design. Latency budgets stay realistic instead of "everything is a flow."

03

Deployment is a database write

Revisions are immutable and content-addressable. Deploy, promote, and rollback are single SQL updates to the deployment table — atomic, zero-downtime, and GitOps-friendly. No file copies, no race conditions.

One platform, designed around what platform teams actually run.

Each surface is a first-class product with its own authoring, deployment, and governance — but they share the same engine, the same tenant model, and the same observability. Pick the surfaces you need today, add the rest later without re-platforming.

FlowJS orchestration with the connectors enterprises actually need.

Architecture

Strict separation. Stateless data plane. Immutable revisions.

Studio writes deployment intent to PostgreSQL. Gateway reads it. There is no synchronous Studio → Gateway runtime dependency — Studio outages never affect API or AI traffic. All gateway state lives in PostgreSQL or Redis, so pods are interchangeable and horizontally scalable.

Security & governance

24 policies. Five stages. One policy engine.

Every API, flow, and AI route runs through the same unified policy engine — stateful policies are cluster-aware when Redis is configured, with local fallback for single-node development. Secrets are referenced, never stored.
